# 01. 제품 기획서 (Product Spec)

> 문서 세트의 시작점. 시스템 설계는 [02 백엔드](./02-backend-spec.md), [03 인증 검증](./03-verification.md), [04 클라이언트](./04-client-architecture.md), [05 디자인](./05-design.md) 참조.

## 1. 문제 정의

등산을 취미로 하는 사람은 "내가 어떤 산을, 어떤 코스로, 몇 번 올랐는지"를 남기고 싶어 한다. 기존 앱은 이 욕구를 **배지·체크리스트**(트랭글, 블랙야크 BAC)나 **트랙 기록**(램블러)으로 풀었다. 그러나 등반 이력이 **지도 위에 누적 채색되는 시각적 자산**으로 쌓이는 경험은 비어 있다.

이 앱의 한 문장 정의:

> **등반한 코스가 지도에 색칠된다. 지도가 곧 나의 등산 이력이다.**

## 2. 경쟁 분석과 차별화 방어

| 앱 | 핵심 기능 | 우리와 겹치는 부분 |
|---|---|---|
| **트랭글** | GPS 트래킹, 전국 1,000여 개 산 정상 도착 시 자동 배지, 경험치/랭킹 | GPS 정상 인증 + 게이미피케이션이 이미 핵심. 배지는 인증샷 없이 자동 발급 |
| **블랙야크 BAC** | 명산100 등 챌린지형 "정복" 프로그램, 위치+사진 정상 인증, 대형 커뮤니티 | "산 정복 인증샷" 문화의 사실상 표준 |
| **램블러** | 등산 특화 트랙 기록, 3,000여 개 코스, 오프라인 지도+배터리 최적화 | 기록 UX의 기준점. 오프라인 대응이 업계 표준임을 보여줌 |
| **KYRO/그라운드 플립** (러닝) | 경로 기반 땅따먹기/영토 점령 | "땅따먹기" 컨셉 자체는 러닝 도메인에서 이미 검증·경쟁 중 |

**정직한 평가**: GPS 정상 인증과 배지는 차별점이 아니다. 남는 차별점은 하나 — **코스 단위로 색칠되는 나만의 정복 지도**. 트랭글은 배지 목록, BAC는 체크리스트 형태이고, 등반 이력이 지도 위에 누적 채색되는 앱은 뚜렷하지 않다.

이 차별점은 표현 계층(presentation-layer)이라 얇다. 그래서 두 가지로 보강한다:

1. **색칠되는 순간을 앱의 핵심 리워드 모먼트로 설계** — [05 §6](./05-design.md) 인증 성공 시퀀스
2. **공유 이미지 카드(v1)** — 색칠된 코스+통계가 SNS 콘텐츠가 되어 바이럴 루프 시작점이 됨 (BAC의 성장 동력이 인증샷 공유 문화였음)

## 3. 페르소나

| | P1. 주말 등산러 (주 타깃) | P2. 챌린지 헌터 | P3. 입문자 |
|---|---|---|---|
| 행태 | 월 1~3회, 수도권 근교 산 | BAC 명산100 병행, 연 30회+ | 친구 따라 첫 등산 |
| 욕구 | 다녀온 산을 남기고 싶다 | 진행률을 한눈에, 자랑하고 싶다 | 어떤 코스가 만만한지 |
| 핵심 가치 | 색칠 지도 = 이력의 시각화 | 카운터·(로드맵) 100대 명산 프리셋 | 난이도 색상 코딩 |

## 4. 핵심 유저 스토리 (v0)

1. 나는 지도에서 내 주변 산과 코스를 난이도 색상으로 볼 수 있다.
2. 나는 산 정상(코스 체크포인트)에서 **탭 한 번**으로 완등을 인증할 수 있다 — 네트워크가 없어도.
3. 인증하면 그 자리에서 코스가 색칠되는 축하 연출을 본다. 서버 확인은 나중에 조용히 끝난다.
4. 나는 기록 탭에서 내가 오른 산 목록과 "나의 n번째 산" 카운터를 본다.

## 5. MVP 범위 — v0 / v1 / v2 컷

> 이 표가 전체 문서의 버전 마킹 기준이다. 각 문서의 `[v1]` `[v2]` 표기는 이 표를 따른다.

| | v0 — 데모 (~6주) | v1 — 포트폴리오 완성 (+4주) | v2 — 실서비스 |
|---|---|---|---|
| **데이터** | 3~5개 산 수동 시딩, checkpoint 종점/수동 지정, 원본 지오메트리 | ETL 파이프라인 + 수도권 20~30산, path_low LOD(서버 zoom 활성) | 100대 명산, 타일 API+CDN, DEM 기반 checkpoint |
| **로그인** | 이메일/비번 + 선가입, users 단일 테이블(provider 컬럼 선반영), 고정 refresh 토큰 | 카카오 로그인, refresh rotation | Apple 로그인, 계정 삭제 API, 강제 업데이트 |
| **완등 인증** | 동기 검증(서버 pending 없음), outbox + 포그라운드 flush(트리거당 1회, 만료 없음), 낙관적 색칠(연출 1개·착지 2개), 관대 정책+flags, 캡처 직후 코스 원탭 선택 | 백그라운드 업로드 세션, 지수 백오프, 만료=flagged 재정의+expired UX, 코스 자동 추천, Android background-task | Play Integrity / App Attest |
| **사진/공유** | 없음 | 사진 1장(스킵 가능) + presigned/lifecycle/EXIF 스트립, 완등 카드(Skia) | 시즌 결산 지도 카드 |
| **지도** | bbox+zoom(서버는 zoom 무시), 클라 타일 양자화 Query 키, 줌 히스테리시스, 저줌 정복 마커 | 200개 상한 유효화, 색칠 시퀀스 목표안 | 서버 타일 + immutable 캐시 |
| **UI** | 탭 2개(지도/기록), 카운터(진행률 링 없음), 도화지 빈 상태+권한 온보딩, 착지 CTA "기록 보기", Sentry | 공유 CTA, 롤백 연출 | 스토어 출시 |
| **API 표면** | `POST /v1/climbs`, `GET /v1/courses?bbox=&zoom=`, `GET /v1/mountains/:id`, `GET /v1/me/climbs`, auth | 사진 presigned 2단계 | 타일 엔드포인트 |

**v0 데모 루프 (이것만으로 닫혀야 함)**: 가입 → 산 보기 → 인증 → 색칠 → **기록 조회**.

### v0 비범위 (명시적 제외)

리더보드, 커뮤니티, 파티, 사진, 공유 카드, 검색 UI(3~5산은 지도 팬으로 충분), 다크 모드, 태블릿.

## 6. 화면 인벤토리 (v0 — 5종)

## 7. 가입 게이트 — v0은 선가입 고정

게스트 둘러보기(가입 지연)는 UX상 매력적이지만, **outbox의 사용자 귀속과 `client_ref` 스코프를 바꾸는 클라이언트 데이터 모델 결정**이다(게스트→가입 승격 시 로컬 초안 마이그레이션 필요). v0은 선가입으로 고정하고, 조회성 API는 public read로 설계해([02 §5](./02-backend-spec.md)) 게스트 전환의 문을 열어둔다. 게스트 모드는 v1 이후 재논의.

## 8. 성공 지표 — 계측과 1:1 매핑

지표는 선언이 아니라 측정 가능해야 한다. **Sentry는 v0 필수 스택**이며, 각 지표는 아래 계측 포인트와 매핑된다.

| 지표 | 정의 | 계측 포인트 (Sentry 이벤트/breadcrumb) |
|---|---|---|
| 인증 성공률 | 캡처 시도 대비 로컬 판정 통과 | 위저드 상태 전이(`acquiring_fix`→각 터미널) |
| 큐 flush 성공률 | flush 시도 대비 서버 2xx | QueueProcessor 결과 |
| 판정 괴리율 | 로컬 통과 후 서버 flagged/rejected 비율 | 서버 응답 `flags`/`status` |
| 로컬-서버 거리 괴리 | 로컬 계산 거리 vs 응답 `distance_m` 차이 | 괴리 큰 경우 이벤트 ([03 §5](./03-verification.md) 단위 버그 가드) |
| 리워드 도달률 | 캡처 성공 대비 시퀀스 완주(착지 CTA 노출) | 시퀀스 완료 이벤트 |
